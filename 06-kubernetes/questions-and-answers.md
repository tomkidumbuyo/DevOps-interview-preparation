## Junior — normal conceptual and code questions

### BRANCH-06-KUBERNETES-JN-01

- [ ] **Question:** What is API groups/versions, and why does it matter in 06 Kubernetes?
> **Covered in:** [Architecture — API groups/versions](01-architecture/README.md#api-groupsversions)

**Answer:** **Definition.** Kubernetes API groups organize related resource kinds and versions define the schema contract served to clients. A served version can differ from the storage version used in etcd, with conversion translating between them. **Mechanism.** The API server resolves group, version and resource from the URL, decodes and validates that schema, defaults fields and converts the object to an internal representation. Before persistence it converts to the configured storage version; reads reverse the conversion for the requested served version. CRDs can use conversion webhooks when schemas differ materially. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Discovery endpoints and CRD/APIService status show what is served; `status.storedVersions` and deprecation metrics reveal migration work. Removing a served version before clients/manifests/controllers migrate causes request failure, while leaving old stored objects can break later conversion or rollback plans. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which API groups/versions matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original 06 Kubernetes journey rather than an administrative green status alone.

### BRANCH-06-KUBERNETES-JN-07

- [ ] **Question:** What is Startup probe, and why does it matter in 06 Kubernetes?
> **Covered in:** [Workloads — Startup probe](02-workloads/README.md#startup-probe)

**Answer:** **Definition.** Delays liveness/readiness scheduling until slow initialization succeeds. **Mechanism.** A caller assigns a deadline and attempt budget, classifies an error, applies bounded backoff with jitter only when retry is safe, and propagates capacity limits upstream. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence shows bounded queues, stable retry ratios and recovery within an objective. Unbounded retries, identical timeouts at every layer, non-idempotent replays, shared resource pools and failover to an unqualified target commonly turn a small fault into an outage. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Startup probe matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original 06 Kubernetes journey rather than an administrative green status alone.
