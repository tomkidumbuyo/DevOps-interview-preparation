## Junior — normal conceptual and code questions

### BRANCH-OPERATIONS-JN-01

- [ ] **Question:** What is kube-state-metrics, and why does it matter in Operations?
> **Covered in:** [Operations — kube-state-metrics](README.md#kube-state-metrics)

**Answer:** **Definition.** Exposes object desired/status metrics rather than container usage. **Mechanism.** Instrumentation observes a defined event or state, attaches bounded context, transports the signal, stores it under retention rules and makes it queryable. A useful signal has documented semantics and can distinguish at least two competing explanations; correlation identifiers belong in logs or traces rather than unbounded metric labels. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence includes coverage of the user journey, known collection delay and actionable SLO or saturation views. Missing telemetry, sampling bias, cardinality explosions, sensitive payload capture and alert thresholds disconnected from user impact are failures of the observability system itself. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which kube-state-metrics matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Operations journey rather than an administrative green status alone.

> **Covered in:** [Operations — Version skew](README.md#version-skew)

**Answer:** **Definition.** Supported kubelet/kube-proxy/controller relationships bound rolling upgrade order. **Mechanism.** The lifecycle begins with versioned intent, validates syntax and policy, resolves dependencies, builds or selects immutable inputs, produces a diff or release plan, changes the target in bounded waves, and records status. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Version skew matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Operations journey rather than an administrative green status alone.

### BRANCH-OPERATIONS-JN-05

- [ ] **Question:** What is etcd snapshot, and why does it matter in Operations?
> **Covered in:** [Operations — etcd snapshot](README.md#etcd-snapshot)

**Answer:** **Definition.** Captures API storage for self-managed control planes with matching encryption/config needs. **Mechanism.** A write normally passes validation and authorization, enters a buffer or transaction, reaches a durable medium, and may then replicate or become visible to readers. Caches and indexes accelerate access but introduce freshness and eviction behavior; snapshots preserve a point-in-time representation but application consistency depends on write ordering. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence combines capacity, latency, error, replication and integrity signals with a tested read or restore. Typical failures include exhausted bytes or inodes, throttling, stale replicas, corrupt metadata, topology mismatch, lost encryption keys and backups that exist but cannot reconstruct the application within RPO/RTO. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which etcd snapshot matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Operations journey rather than an administrative green status alone.

### BRANCH-OPERATIONS-JN-08

- [ ] **Code:** **Question:** What does `kubectl describe pod POD -n NS` help you verify for Operations?
> **Covered in:** [Operations — Worked command and configuration example](README.md#worked-command-and-configuration-example)

**Answer:** **Purpose.** `kubectl describe pod POD -n NS` reads Kubernetes API desired/status state or recent reconciliation evidence without treating a single `Ready` value as proof of the user journey. **Mechanism.** The relevant Operations concept is Events: Scheduling, pull, mount, probe and controller records point to the first failing transition. The command reads one representation of desired or observed state; it does not by itself prove that every controller, dependency or user path is healthy. **Evidence.** Confirm identity, account/project/cluster, location and exact resource first, then preserve IDs, status/reason fields, timestamps and exit status. Compare them with source-controlled intent and a known healthy baseline, and correlate the nearest event, metric, log, trace or audit record. **Failure and trade-off.** Empty output, permission denial, stale state and an unavailable dependency require different next checks; immediately mutating the target destroys evidence and may widen impact. **Production example.** During a Operations rollout, an operator uses this check to decide whether the fault is in accepted configuration, reconciliation or the live data path, and verifies the final repair from the original client.
