# Architecture — interview questions and answers

> Study first: [README.md](README.md) · Minimum: 20 answered questions per level, split into normal and procedural. Code/CLI prompts are marked **Code**.

## Junior — normal conceptual and code questions

### BRANCH-ARCHITECTURE-JN-01

- [ ] **Question:** What is API groups/versions, and why does it matter in Architecture?
> **Covered in:** [Architecture — API groups/versions](README.md#api-groupsversions)

**Answer:** **Definition.** Kubernetes API groups organize related resource kinds and versions define the schema contract served to clients. A served version can differ from the storage version used in etcd, with conversion translating between them. **Mechanism.** The API server resolves group, version and resource from the URL, decodes and validates that schema, defaults fields and converts the object to an internal representation. Before persistence it converts to the configured storage version; reads reverse the conversion for the requested served version. CRDs can use conversion webhooks when schemas differ materially. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Discovery endpoints and CRD/APIService status show what is served; `status.storedVersions` and deprecation metrics reveal migration work. Removing a served version before clients/manifests/controllers migrate causes request failure, while leaving old stored objects can break later conversion or rollback plans. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which API groups/versions matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Architecture journey rather than an administrative green status alone.

### BRANCH-ARCHITECTURE-JN-02

- [ ] **Question:** What is Request path, and why does it matter in Architecture?
> **Covered in:** [Architecture — Request path](README.md#request-path)

**Answer:** **Definition.** The Kubernetes API request path is the ordered chain that turns an HTTPS request into an authorized, admitted and persisted object or a precise rejection. **Mechanism.** After TLS termination and request parsing, authentication establishes user and groups; authorization evaluates the verb/resource/name/namespace; mutating admission can change the object; schema defaulting and validation establish a valid representation; validating admission enforces final policy; the registry layer applies concurrency and persistence through etcd. Audit stages can record receipt, response start and completion. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** HTTP status, audit events, admission messages, API latency histograms and etcd metrics localize failure. Webhook timeouts or broad failure-closed policies can block unrelated writes, while failure-open can bypass policy, so webhook scope, timeout and availability are production design decisions. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Request path matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Architecture journey rather than an administrative green status alone.

### BRANCH-ARCHITECTURE-JN-03

- [ ] **Question:** What is Controller loop, and why does it matter in Architecture?
> **Covered in:** [Architecture — Controller loop](README.md#controller-loop)

**Answer:** **Definition.** Watches desired/observed state and performs level-based idempotent reconciliation. **Mechanism.** The mechanism receives configuration or runtime input, validates preconditions, changes or derives state, and exposes status to the next component. Synchronous parts return a direct outcome; asynchronous parts need durable status, reconciliation and idempotency because the original caller may disappear. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence shows the expected state transition and its owner, timestamp and reason. Invalid input, stale state, conflicting writers, hidden limits and dependencies that partially complete are the most useful failure categories to distinguish. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Controller loop matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Architecture journey rather than an administrative green status alone.

### BRANCH-ARCHITECTURE-JN-04

- [ ] **Question:** What is Scheduler queue, and why does it matter in Architecture?
> **Covered in:** [Architecture — Scheduler queue](README.md#scheduler-queue)

**Answer:** **Definition.** Unscheduled Pods progress through filtering/scoring/backoff before binding. **Mechanism.** A producer serializes and publishes a message, the service stores or routes it, a consumer receives it under a delivery contract, and acknowledgement advances or removes the item. At-least-once systems can redeliver after an unknown outcome, so stable operation identifiers and idempotent state changes are part of correctness. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence includes publish and consume rates, queue depth and age, processing latency, retry/dead-letter counts and end-to-end business completion. Poison messages, hot partitions, backlog growth, duplicate effects and retry storms are common; adding consumers helps only when the downstream dependency and partition model can absorb them. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Scheduler queue matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Architecture journey rather than an administrative green status alone.
