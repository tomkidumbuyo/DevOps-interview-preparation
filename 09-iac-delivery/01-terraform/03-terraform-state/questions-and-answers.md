## Junior — normal conceptual and code questions

### TERRAFORM-STATE-JN-01

- [ ] **Question:** What is State purpose, and why does it matter in Terraform state?
> **Covered in:** [Terraform state — State purpose](README.md#state-purpose)

**Answer:** **Definition.** The term State purpose refers to a configuration, state or promotion mechanism that moves a reviewed revision toward effective runtime state within Terraform state. **Mechanism.** A write normally passes validation and authorization, enters a buffer or transaction, reaches a durable medium, and may then replicate or become visible to readers. Caches and indexes accelerate access but introduce freshness and eviction behavior; snapshots preserve a point-in-time representation but application consistency depends on write ordering. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence combines capacity, latency, error, replication and integrity signals with a tested read or restore. Typical failures include exhausted bytes or inodes, throttling, stale replicas, corrupt metadata, topology mismatch, lost encryption keys and backups that exist but cannot reconstruct the application within RPO/RTO. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which State purpose matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Terraform state journey rather than an administrative green status alone.

- [ ] **Code:** **Question:** What does `git diff --check` help you verify for Terraform state?
> **Covered in:** [Terraform state — State purpose](README.md#state-purpose)

**Answer:** **Purpose.** `git diff --check` inspects the repository's object graph, refs or diff so a delivery decision is tied to exact content rather than an assumed branch name. **Mechanism.** The relevant Terraform state concept is State recovery: Is a controlled state transition requiring inventory, compatibility, protected state, rehearsal, rollback/abort criteria, integrity checks and measured user-facing RPO/RTO or completion. The command reads one representation of desired or observed state; it does not by itself prove that every controller, dependency or user path is healthy. **Evidence.** Confirm identity, account/project/cluster, location and exact resource first, then preserve IDs, status/reason fields, timestamps and exit status. Compare them with source-controlled intent and a known healthy baseline, and correlate the nearest event, metric, log, trace or audit record. **Failure and trade-off.** Empty output, permission denial, stale state and an unavailable dependency require different next checks; immediately mutating the target destroys evidence and may widen impact. **Production example.** During a Terraform state rollout, an operator uses this check to decide whether the fault is in accepted configuration, reconciliation or the live data path, and verifies the final repair from the original client.
