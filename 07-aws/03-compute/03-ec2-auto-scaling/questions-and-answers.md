## Junior — normal conceptual and code questions

### EC2-AUTO-SCALING-JN-01

- [ ] **Question:** What is Launch template, and why does it matter in EC2 Auto Scaling?
> **Covered in:** [EC2 Auto Scaling — Launch template](README.md#launch-template)

**Answer:** **Definition.** Versions instance image/type/network/role/storage/bootstrap inputs. **Mechanism.** The mechanism receives configuration or runtime input, validates preconditions, changes or derives state, and exposes status to the next component. Synchronous parts return a direct outcome; asynchronous parts need durable status, reconciliation and idempotency because the original caller may disappear. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence shows the expected state transition and its owner, timestamp and reason. Invalid input, stale state, conflicting writers, hidden limits and dependencies that partially complete are the most useful failure categories to distinguish. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Launch template matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original EC2 Auto Scaling journey rather than an administrative green status alone.

The command reads one representation of desired or observed state; it does not by itself prove that every controller, dependency or user path is healthy. **Evidence.** Confirm identity, account/project/cluster, location and exact resource first, then preserve IDs, status/reason fields, timestamps and exit status. Compare them with source-controlled intent and a known healthy baseline, and correlate the nearest event, metric, log, trace or audit record. **Failure and trade-off.** Empty output, permission denial, stale state and an unavailable dependency require different next checks; immediately mutating the target destroys evidence and may widen impact. **Production example.** During a EC2 Auto Scaling rollout, an operator uses this check to decide whether the fault is in accepted configuration, reconciliation or the live data path, and verifies the final repair from the original client.

> **Covered in:** [EC2 Auto Scaling — Launch template](README.md#launch-template)

Min/max/desired: Bound fleet capacity while policies adjust desired count. **Mechanism.** In EC2 Auto Scaling, compare where each acts in the request or resource lifecycle, which state it owns, whether its result is synchronous or eventually reconciled, and how callers observe completion. They may be complementary layers rather than substitutes. **Decision criteria.** Select using workload shape and protocol, identity and tenant isolation, failure-domain independence, consistency and recovery contract, latency/throughput limits, team ownership and total cost. **Failure evidence.** State which metric, event, status field or controlled test would distinguish failure of one from the other, and how rollback differs. **Production example.** For a production EC2 Auto Scaling design, use the simpler option until measured SLO, isolation, scale or recovery evidence crosses an explicit threshold; document that threshold so the decision can be revisited instead of becoming permanent folklore.
