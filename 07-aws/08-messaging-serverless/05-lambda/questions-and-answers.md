# AWS Lambda — interview questions and answers

## Junior — normal conceptual and code questions

### AWS-LAMBDA-JN-01

- [ ] **Question:** What is Execution environment, and why does it matter in AWS Lambda?
> **Covered in:** [AWS Lambda — Execution environment](README.md#execution-environment)

**Answer:** **Definition.** Initialization may be reused across invocations but must not hold unsafe tenant state. **Mechanism.** The mechanism receives configuration or runtime input, validates preconditions, changes or derives state, and exposes status to the next component. Synchronous parts return a direct outcome; asynchronous parts need durable status, reconciliation and idempotency because the original caller may disappear. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence shows the expected state transition and its owner, timestamp and reason. Invalid input, stale state, conflicting writers, hidden limits and dependencies that partially complete are the most useful failure categories to distinguish. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Execution environment matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original AWS Lambda journey rather than an administrative green status alone.

### AWS-LAMBDA-JN-07

- [ ] **Question:** What is VPC networking, and why does it matter in AWS Lambda?
> **Covered in:** [AWS Lambda — VPC networking](README.md#vpc-networking)

**Answer:** **Definition.** Subnets, IP capacity, SG, NAT/endpoints/DNS become function dependencies. **Mechanism.** A real request crosses several independent states: name resolution returns an address, the source selects a route and source address, link or overlay forwarding reaches the next hop, stateful or stateless policy evaluates the flow, transport establishes communication, and TLS/application protocols negotiate their own contract. The return path must also work. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence progresses layer by layer: correct name/address, expected route, permitted flow, listening endpoint, successful handshake and valid application response. Timeouts, refusals, resets and protocol errors mean different layers; packet loss, MTU, asymmetric paths, connection-state exhaustion and proxy timeout mismatch are common production failures. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which VPC networking matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original AWS Lambda journey rather than an administrative green status alone.

### AWS-LAMBDA-JN-08

- [ ] **Code:** **Question:** What does `aws logs tail /aws/lambda/FUNCTION --since 30m --follow` help you verify for AWS Lambda?
> **Covered in:** [AWS Lambda — Worked command and configuration example](README.md#worked-command-and-configuration-example)

**Answer:** **Purpose.** `aws logs tail /aws/lambda/FUNCTION --since 30m --follow` queries the AWS service control plane; IDs, Region, status/reason fields and request attribution should be correlated with CloudTrail and service metrics. **Mechanism.** The relevant AWS Lambda concept is Execution role: Function AWS permissions should exclude deployment/administrative capabilities. The command reads one representation of desired or observed state; it does not by itself prove that every controller, dependency or user path is healthy. **Evidence.** Confirm identity, account/project/cluster, location and exact resource first, then preserve IDs, status/reason fields, timestamps and exit status. Compare them with source-controlled intent and a known healthy baseline, and correlate the nearest event, metric, log, trace or audit record. **Failure and trade-off.** Empty output, permission denial, stale state and an unavailable dependency require different next checks; immediately mutating the target destroys evidence and may widen impact. **Production example.** During a AWS Lambda rollout, an operator uses this check to decide whether the fault is in accepted configuration, reconciliation or the live data path, and verifies the final repair from the original client.
