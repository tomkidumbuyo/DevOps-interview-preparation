### GUARDDUTY-SECURITY-HUB-INSPECTOR-MACIE-AND-DETEC-JN-01

- [ ] **Question:** What is GuardDuty, and why does it matter in GuardDuty, Security Hub, Inspector, Macie and Detective?
> **Covered in:** [GuardDuty, Security Hub, Inspector, Macie and Detective — GuardDuty](README.md#guardduty)

**Answer:** **Definition.** Analyzes supported logs/signals to emit contextual threat findings. **Mechanism.** The mechanism receives configuration or runtime input, validates preconditions, changes or derives state, and exposes status to the next component. Synchronous parts return a direct outcome; asynchronous parts need durable status, reconciliation and idempotency because the original caller may disappear. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence shows the expected state transition and its owner, timestamp and reason. Invalid input, stale state, conflicting writers, hidden limits and dependencies that partially complete are the most useful failure categories to distinguish. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which GuardDuty matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original GuardDuty, Security Hub, Inspector, Macie and Detective journey rather than an administrative green status alone.

### GUARDDUTY-SECURITY-HUB-INSPECTOR-MACIE-AND-DETEC-JN-02

- [ ] **Question:** What is Security Hub, and why does it matter in GuardDuty, Security Hub, Inspector, Macie and Detective?
> **Covered in:** [GuardDuty, Security Hub, Inspector, Macie and Detective — Security Hub](README.md#security-hub)

**Answer:** **Definition.** Aggregates/normalizes findings and control standards across accounts/Regions. **Mechanism.** At runtime a caller presents or derives an identity, the enforcement point gathers identity, resource and request context, and applicable rules produce allow or deny. The effective decision is the intersection of all guardrails; encryption protects bytes but does not replace authorization, and audit records explain which decision was made. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence includes a short-lived attributable identity, narrowly scoped access and an audit event for the intended resource. Failures commonly come from expired credentials, mismatched trust, an overriding deny, wrong resource scope or key/certificate lifecycle problems; widening access may hide the cause while creating a breach path. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Security Hub matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original GuardDuty, Security Hub, Inspector, Macie and Detective journey rather than an administrative green status alone.

**Answer:** **Purpose.** `aws macie2 get-macie-session` queries the AWS service control plane; IDs, Region, status/reason fields and request attribution should be correlated with CloudTrail and service metrics. **Mechanism.** The relevant GuardDuty, Security Hub, Inspector, Macie and Detective concept is Suppression: Tuned archival/suppression needs owner/reason/expiry to avoid hiding new risk. The command reads one representation of desired or observed state; it does not by itself prove that every controller, dependency or user path is healthy. **Evidence.** Confirm identity, account/project/cluster, location and exact resource first, then preserve IDs, status/reason fields, timestamps and exit status. Compare them with source-controlled intent and a known healthy baseline, and correlate the nearest event, metric, log, trace or audit record. **Failure and trade-off.** Empty output, permission denial, stale state and an unavailable dependency require different next checks; immediately mutating the target destroys evidence and may widen impact. **Production example.** During a GuardDuty, Security Hub, Inspector, Macie and Detective rollout, an operator uses this check to decide whether the fault is in accepted configuration, reconciliation or the live data path, and verifies the final repair from the original client.

## Mid-level — normal, comparison and configuration questions

### GUARDDUTY-SECURITY-HUB-INSPECTOR-MACIE-AND-DETEC-MN-01

- [ ] **Question:** Compare GuardDuty with Security Hub. When would each change your design?
> **Covered in:** [GuardDuty, Security Hub, Inspector, Macie and Detective — Security Hub](README.md#security-hub)

**Answer:** **Definitions.** GuardDuty: Analyzes supported logs/signals to emit contextual threat findings. Security Hub: Aggregates/normalizes findings and control standards across accounts/Regions. **Mechanism.** In GuardDuty, Security Hub, Inspector, Macie and Detective, compare where each acts in the request or resource lifecycle, which state it owns, whether its result is synchronous or eventually reconciled, and how callers observe completion. They may be complementary layers rather than substitutes. **Decision criteria.** Select using workload shape and protocol, identity and tenant isolation, failure-domain independence, consistency and recovery contract, latency/throughput limits, team ownership and total cost. **Failure evidence.** State which metric, event, status field or controlled test would distinguish failure of one from the other, and how rollback differs. **Production example.** For a production GuardDuty, Security Hub, Inspector, Macie and Detective design, use the simpler option until measured SLO, isolation, scale or recovery evidence crosses an explicit threshold; document that threshold so the decision can be revisited instead of becoming permanent folklore.
