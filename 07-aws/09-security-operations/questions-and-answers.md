## Junior — normal conceptual and code questions

### BRANCH-SECURITY-OPERATIONS-JN-01

- [ ] **Question:** What is KMS key, and why does it matter in Security Operations?
> **Covered in:** [Security Operations — KMS key](README.md#kms-key)

**Answer:** **Definition.** Logical key with policy, metadata and protected key material used by integrated services. **Mechanism.** At runtime a caller presents or derives an identity, the enforcement point gathers identity, resource and request context, and applicable rules produce allow or deny. The effective decision is the intersection of all guardrails; encryption protects bytes but does not replace authorization, and audit records explain which decision was made. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence includes a short-lived attributable identity, narrowly scoped access and an audit event for the intended resource. Failures commonly come from expired credentials, mismatched trust, an overriding deny, wrong resource scope or key/certificate lifecycle problems; widening access may hide the cause while creating a breach path. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which KMS key matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Security Operations journey rather than an administrative green status alone.

### BRANCH-SECURITY-OPERATIONS-JN-06

- [ ] **Question:** What is DNS validation, and why does it matter in Security Operations?
> **Covered in:** [Security Operations — DNS validation](README.md#dns-validation)

**Answer:** **Definition.** Persistent validation record enables managed renewal when service conditions hold. **Mechanism.** A real request crosses several independent states: name resolution returns an address, the source selects a route and source address, link or overlay forwarding reaches the next hop, stateful or stateless policy evaluates the flow, transport establishes communication, and TLS/application protocols negotiate their own contract. The return path must also work. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence progresses layer by layer: correct name/address, expected route, permitted flow, listening endpoint, successful handshake and valid application response. Timeouts, refusals, resets and protocol errors mean different layers; packet loss, MTU, asymmetric paths, connection-state exhaustion and proxy timeout mismatch are common production failures. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which DNS validation matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Security Operations journey rather than an administrative green status alone.

### BRANCH-SECURITY-OPERATIONS-JN-08

- [ ] **Code:** **Question:** What does `aws kms describe-key --key-id KEY` help you verify for Security Operations?
> **Covered in:** [Security Operations — KMS key](README.md#kms-key)

**Answer:** **Purpose.** `aws kms describe-key --key-id KEY` queries the AWS service control plane; IDs, Region, status/reason fields and request attribution should be correlated with CloudTrail and service metrics. **Mechanism.** The relevant Security Operations concept is Security Hub: Aggregates/normalizes findings and control standards across accounts/Regions. The command reads one representation of desired or observed state; it does not by itself prove that every controller, dependency or user path is healthy. **Evidence.** Confirm identity, account/project/cluster, location and exact resource first, then preserve IDs, status/reason fields, timestamps and exit status. Compare them with source-controlled intent and a known healthy baseline, and correlate the nearest event, metric, log, trace or audit record. **Failure and trade-off.** Empty output, permission denial, stale state and an unavailable dependency require different next checks; immediately mutating the target destroys evidence and may widen impact. **Production example.** During a Security Operations rollout, an operator uses this check to decide whether the fault is in accepted configuration, reconciliation or the live data path, and verifies the final repair from the original client.
