## Junior — normal conceptual and code questions

### AWS-CDK-JN-01

- [ ] **Question:** What is App/stack/construct tree, and why does it matter in AWS CDK?
> **Covered in:** [AWS CDK — App/stack/construct tree](README.md#appstackconstruct-tree)

**Answer:** **Definition.** Code composes logical resources and ownership hierarchy. **Mechanism.** The mechanism receives configuration or runtime input, validates preconditions, changes or derives state, and exposes status to the next component. Synchronous parts return a direct outcome; asynchronous parts need durable status, reconciliation and idempotency because the original caller may disappear. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Healthy evidence shows the expected state transition and its owner, timestamp and reason. Invalid input, stale state, conflicting writers, hidden limits and dependencies that partially complete are the most useful failure categories to distinguish. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which App/stack/construct tree matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original AWS CDK journey rather than an administrative green status alone.

### AWS-CDK-JN-08

- [ ] **Code:** **Question:** What does `cdk doctor` help you verify for AWS CDK?
> **Covered in:** [AWS CDK — Worked command and configuration example](README.md#worked-command-and-configuration-example)

**Answer:** **Purpose.** `cdk doctor` captures a read-oriented state snapshot that must be interpreted against a healthy baseline, the exact target and the next adjacent dependency. **Mechanism.** The relevant AWS CDK concept is Escape hatch: Overrides low-level property when abstraction lacks it, increasing coupling. The command reads one representation of desired or observed state; it does not by itself prove that every controller, dependency or user path is healthy. **Evidence.** Confirm identity, account/project/cluster, location and exact resource first, then preserve IDs, status/reason fields, timestamps and exit status. Compare them with source-controlled intent and a known healthy baseline, and correlate the nearest event, metric, log, trace or audit record. **Failure and trade-off.** Empty output, permission denial, stale state and an unavailable dependency require different next checks; immediately mutating the target destroys evidence and may widen impact. **Production example.** During a AWS CDK rollout, an operator uses this check to decide whether the fault is in accepted configuration, reconciliation or the live data path, and verifies the final repair from the original client.
