# Memory — interview questions and answers

> Study first: [README.md](README.md) · Minimum: 20 answered questions per level, split into normal and procedural. Code/CLI prompts are marked **Code**.

## Junior — normal conceptual and code questions

### MEMORY-JN-01

- [ ] **Question:** What is Virtual memory, and why does it matter in Memory?
> **Covered in:** [Memory — Virtual memory](README.md#virtual-memory)

**Answer:** **Definition.** Virtual memory is the per-process address space presented by the kernel. A virtual address is not a RAM location; page tables map it to a physical frame, a file-backed page, a shared frame, a swapped-out page or no page at all. **Mechanism.** When code reserves memory, the kernel usually creates a virtual mapping without immediately allocating every physical page. First access raises a page fault; the kernel validates the mapping, obtains or loads a page, updates the page table and resumes the instruction. The MMU and TLB accelerate later translations. Copy-on-write lets processes initially share a page and creates a private copy only after a write. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** `VSZ` can therefore be much larger than RAM without indicating a leak. Investigate RSS, proportional set size, anonymous versus file-backed pages, minor and major faults, swap and cgroup limits. Address-space exhaustion, permission faults, TLB pressure and overcommit/OOM behavior are distinct failure modes. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Virtual memory matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Memory journey rather than an administrative green status alone.

### MEMORY-JN-02

- [ ] **Question:** What is Resident memory, and why does it matter in Memory?
> **Covered in:** [Memory — Resident memory](README.md#resident-memory)

**Answer:** **Definition.** Resident memory is the subset of a process's mapped pages currently present in physical RAM. RSS counts resident pages but can double-count shared pages across processes, so it is not the same as uniquely owned memory. **Mechanism.** The kernel updates mappings as pages are faulted in and reclaimed. Anonymous heap/stack, shared libraries, memory-mapped files and page-cache-backed mappings can all contribute to RSS. Proportional set size divides shared pages between users and is often better for estimating a process's real contribution. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** A healthy service has an RSS distribution compatible with its workload and cgroup limit. A steadily growing anonymous RSS, rising reclaim stalls or repeated cgroup `oom_kill` events is stronger leak or capacity evidence than one large VSZ value. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Resident memory matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Memory journey rather than an administrative green status alone.

### MEMORY-JN-03

- [ ] **Question:** What is Shared memory, and why does it matter in Memory?
> **Covered in:** [Memory — Shared memory](README.md#shared-memory)

**Answer:** **Definition.** Shared memory maps the same physical pages into more than one process so processes can exchange data without copying it through a socket or file for every operation. POSIX shared memory, System V shared memory and shared `mmap` mappings provide related interfaces. **Mechanism.** The kernel creates a shared object or file-backed mapping and installs references to its pages in each participating process's page tables. The CPU cache-coherence system makes memory updates visible, while the application must use mutexes, semaphores, atomics or another protocol for ordering and consistency. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Shared memory reduces copy and serialization cost but expands the corruption and synchronization blast radius. Inspect mapping size, ownership and permissions plus application lock contention; stale segments, permissive modes, false sharing and a crashed writer leaving inconsistent data are common failures. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Shared memory matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Memory journey rather than an administrative green status alone.

### MEMORY-JN-04

- [ ] **Question:** What is Page cache, and why does it matter in Memory?
> **Covered in:** [Memory — Page cache](README.md#page-cache)

**Answer:** **Definition.** The page cache is RAM used by the Linux kernel to cache file contents. Reads can be served without another device access, and buffered writes modify cached pages before writeback makes them durable. **Mechanism.** A file read faults or copies pages into cache; later reads reuse them. A buffered write marks pages dirty, background writeback flushes them according to kernel thresholds, and `fsync` or related application ordering asks for durability under the storage contract. Clean cache is readily reclaimable, while dirty-page pressure can force foreground work to wait. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** High cache usage is normally healthy. Diagnose cache hit behavior, dirty/writeback pages, major faults, I/O latency and PSI together. Dropping caches in production destroys useful state and usually hides the real workload or storage bottleneck. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Page cache matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Memory journey rather than an administrative green status alone.

### MEMORY-JN-05

- [ ] **Question:** What is Buffers, and why does it matter in Memory?
> **Covered in:** [Memory — Buffers](README.md#buffers)

**Answer:** **Definition.** In modern Linux memory reporting, buffers are primarily kernel memory used for block-device metadata and related I/O bookkeeping, while cached file contents are accounted mainly as page cache. The historical `buffers/cache` grouping is why the terms are often confused. **Mechanism.** Kernel subsystems allocate buffer heads or other metadata structures to describe block mappings and I/O. These structures participate in reclaim and are reported separately or inside broader reclaimable slab/cache counters depending on the kernel and tool. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Do not diagnose a leak from the `buff/cache` column alone. Use `/proc/meminfo`, slab information, reclaim and workload evidence to separate file cache, reclaimable kernel objects and unreclaimable slab growth. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Buffers matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Memory journey rather than an administrative green status alone.

### MEMORY-JN-06

- [ ] **Question:** What is Swap, and why does it matter in Memory?
> **Covered in:** [Memory — Swap](README.md#swap)

**Answer:** **Definition.** Swap is disk- or device-backed space that can hold eligible anonymous memory pages evicted from RAM. It extends reclaim choices but is much slower than RAM and is not a substitute for sufficient capacity. **Mechanism.** Under pressure, the kernel selects cold anonymous pages, writes them to a swap area and updates their page-table entries. Access later causes a major fault and reads the page back. `swappiness`, memory cgroups, workload locking and platform policy influence use; file-backed clean pages can usually be dropped and reread instead. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Some swap use can be harmless after an old pressure event. Sustained `si/so`, major faults and PSI with user latency indicates thrashing. Disabling swap changes failure behavior toward earlier OOM and may be required or discouraged by specific orchestration versions, so treat it as a workload/platform decision. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Swap matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Memory journey rather than an administrative green status alone.

### MEMORY-JN-07

- [ ] **Question:** What is Memory pressure, and why does it matter in Memory?
> **Covered in:** [Memory — Memory pressure](README.md#memory-pressure)

**Answer:** **Definition.** Memory pressure is the condition in which workloads spend time reclaiming, compacting or waiting for memory because readily usable capacity is scarce. It is about stalled useful work, not merely a low `MemFree` value. **Mechanism.** Allocation crosses watermarks and wakes background reclaim; heavier demand enters direct reclaim in the allocating task. The kernel drops cache, writes dirty pages, swaps eligible anonymous pages or compacts memory for contiguous allocations. If progress is impossible inside a host or cgroup boundary, OOM handling begins. Its guarantee applies only at its own boundary, so upstream identity, downstream dependencies and asynchronous convergence still matter. **Evidence and failure behavior.** Use memory PSI, reclaim scans, major faults, swap, dirty/writeback pages, cgroup `memory.events` and latency. A full page cache with low stalls is healthy; rising `some/full` PSI, allocation latency and OOM events is actionable pressure. Compare those signals with desired configuration, workload shape and a known healthy baseline before choosing a mitigation. **Trade-off.** The choice normally balances simplicity, isolation, performance, recovery effort, portability and cost; moving a boundary solves one problem while shifting another to the caller or operator. **Production example.** A team operating a service in which Memory pressure matters records the workload and configuration baseline, injects one bounded failure in a sandbox, alerts on the first user-impact or saturation signal, and verifies recovery through the original Memory journey rather than an administrative green status alone.
